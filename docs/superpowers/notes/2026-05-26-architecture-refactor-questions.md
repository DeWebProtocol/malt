# Architecture Refactor Questions

This note records design questions discovered while refactoring toward the entrypoint-driven architecture:

- `malt add --target malt` can avoid CLI-side materialization only if the daemon has a write API that supports an undefined or newly-created UnixFS root. The current `POST /{root}/{path}` route requires a root CID in the path, while the existing CLI can create a fresh root locally via `CreateRootStructure`.
- The existing CLI staged add path supports merging a local add tree with an existing root before materialization. A direct per-file daemon replacement must preserve prefix/wrap/ignore/symlink semantics or intentionally narrow them; the current refactor instead moved the staged planner into `layout/unixfs`.
- Symlink directory boundaries are intentionally preserved as separate staged map roots because existing CLI contracts and tests depend on that boundary. The implementation now preserves it through `layout/unixfs` staged materialization rather than CLI-owned tree semantics.
- The server still has legacy stat/migration paths for pre-`layout/unixfs` structures. Those paths should be kept until old roots are either migrated or declared unsupported.
- Add summary counters remain computed from staged layout materialization receipts and CAS batcher observations. A future daemon batch-add API would need richer receipts if this accounting moved fully server-side.
- The public `sdk/client.AddUnixFSFile` still accepts `[]byte`; the CLI staged add path no longer uses it for large files and instead streams file readers through `layout/unixfs.MaterializeStagedFilePayload`.
- `server.registerRoutes` previously registered `POST /{root}/_batch-update` even though `Handler` already catches removed routes before `ServeMux`. The duplicate mux registration conflicts with fresh-root UnixFS route patterns and appears redundant.
- Fresh-root UnixFS writes use `POST /_unixfs?path=...` instead of a catch-all route because Go `ServeMux` rejects overlaps with root-scoped writer routes such as `/{root}/_mutate`.
- `sdk/client.unixFSWriteRoute` now returns a mutable query map for both fresh-root and root-scoped writes so directory writes can append `type=dir` uniformly.
- Directly switching `malt add --target malt` to per-file daemon UnixFS writes breaks existing CLI contracts: symlink-directory map boundaries, CAS batch usage, duplicate small-payload dedupe, and existing-root symlink replacement behavior. The implemented shared `layout/unixfs` staged planner preserves these contracts without adding a daemon batch API.
- Added a server regression test for `POST /_unixfs?path=...` to lock the fresh-root daemon/layout path.
- The old `cmd/malt/uploadAsList` split is removed. Fixed-list chunk upload, temporary base root creation, mutation construction, and mutation application are now delegated through `layout/unixfs.MaterializeFixedListPayload` and the sdk fixed-list writer port.
- Server target stat conversion is now centralized in `pathStatForTarget`; legacy raw-target fallback still intentionally tolerates missing CAS data when compatibility callers already resolved a target CID.
- Directory manifest JSON creation now goes through the top-level `layout/unixfs.DirectoryManifestPayload` facade; staged materialization lives in `layout/unixfs` and no longer hand-rolls manifest payload serialization in CLI code.
- CID-to-storage-kind projection now lives in `layout/unixfs/internal/format.StorageKindFromCID`; CLI add tree construction no longer owns the codec mapping.
- CLI and server fixed chunk-size constants now reference `layout/unixfs.DefaultChunkSize`; upper layers no longer duplicate the numeric layout default.
- `layout/unixfs.Stat` now uses `layout/unixfs/internal/format.StorageKindFromCID`; the codec-to-storage-kind mapping has one implementation.
- Directory root create-structure bindings now go through `layout/unixfs/internal/format.DirectoryRootBindings`; CLI materialization no longer hand-assembles `@payload` plus flattened descendant bindings.
- Create-structure binding counting now uses `layout/unixfs/internal/format.CountDefinedBindings`; CLI no longer owns binding-map accounting details.
- `cmd/malt/add_materialize.go` now only adapts daemon `Stat` and `CreateRootStructure` into layout interfaces; existing-tree loading and binding-count ownership moved to `layout/unixfs`.
- Payload chunk splitting now lives in `layout/unixfs.PayloadChunks` and `layout/unixfs.StreamPayloadChunks`; CLI upload code and layout commit code share the same chunking rule.
- List payload range proof selection now lives in `layout/unixfs.AppendListPayloadRangeProof`; server no longer branches between measured-list range evidence and composed list-index evidence.
- ProofList list-index and measured-list verification now lives in `layout/unixfs.VerifyProofListListStructure`; server keeps only ProofList traversal policy and delegates UnixFS list proof semantics to the layout package.
- Remaining `server` references to `writer.ArcSetDelta` are part of the generic HTTP mutation DTO adapter, not UnixFS application semantics. I did not move them because the server still needs to translate public API JSON into writer mutations.
- Directory manifest payload reading now lives behind `layout/unixfs.ManifestDirectoryEntries` and `layout/unixfs.DirectoryManifestPayloadEntries`; server no longer imports UnixFS manifest/wire internals for stat or legacy migration.
- CLI add materialization now uses the top-level `layout/unixfs` facade for manifest payloads, storage-kind projection, directory root bindings, and binding counts; production entrypoints no longer import `layout/unixfs/internal/manifest` or `layout/unixfs/internal/format` directly.
- Added a source-layout guard requiring server and CLI production entrypoints to use the UnixFS facade instead of the manifest/wire subpackages.
- Server no longer passes the UnixFS default chunk size back into payload-level list helpers; `layout/unixfs.Layout` owns that setting and exposes `ListPayloadSize`, `ReadListPayloadRange`, and `AppendListPayloadRangeProof` without a caller-supplied chunk-size parameter.
- The top-level `layout/unixfs` facade now also exposes manifest codec/CID helpers. Server, SDK, and CLI tests use the facade instead of importing `layout/unixfs/internal/format` or `layout/unixfs/internal/manifest` directly, leaving those subpackages as layout-internal implementation details plus their own unit tests.
- Server verifier traversal state and explicit evidence decoding now live in `service_verify.go`; generic semantic mutation request parsing now lives in `semantic_mutation_request.go`. This leaves `handlers.go` focused on shared HTTP/content/UnixFS helpers instead of verifier and write-DTO adapters.
- CLI staged tree shape, path normalization, file/mapdir node updates, and merge logic now live in `layout/unixfs.StagedNode` helpers. The old `cmd/malt/add_tree.go` wrapper file has been deleted.
- Staged directory materialization and existing-root staged-tree loading now live in `layout/unixfs.MaterializeStagedDirectory` and `layout/unixfs.LoadStagedCurrentTree`. `cmd/malt/add_materialize.go` now only adapts daemon `CreateRootStructure`/`Stat` into layout interfaces.
- Fixed-list payload materialization now lives in `layout/unixfs.MaterializeFixedListPayload`; `sdk/client.Client` implements the fixed-list writer port. CLI no longer has a separate `uploadAsList` helper.
- Added source-layout guards that prevent CLI add from reintroducing direct fixed-list mutation construction, payload-root/mutation endpoint calls, or hand-rolled manifest/binding materialization.
- Staged file payload policy now lives in `layout/unixfs.MaterializeStagedFilePayload`; CLI no longer decides raw-vs-list payload layout and no longer has a separate `uploadAsList` path.
