# MALT v0.0.3

Status: released on 2026-07-12

`v0.0.3` packages MALT as a general arc-granularity graph
data-authentication library. It is an experimental source release, not a
stable API line or a production managed gateway.

The source tree was validated first as `v0.0.3-rc.1`, then published from the
same approved commit as the final `v0.0.3` tag. The historical
`v0.0.3-core-boundary` milestone is not a release tag.

## Highlights

- The module-root `malt` package exposes typed `Query`, `ReadRequest`,
  `ReadResult`, and `Engine.Read`/`Apply`/`VerifyRead` contracts.
- `auth/verifier` verifies real radix-map, tree-list, and measured-range
  ProofList evidence without ArcTable, CAS, runtime, layout, server, daemon, or
  network access. Built-in portable backends cover KZG and IPA typed roots.
- `graph/verifier` remains a thin reference-runtime adapter over the portable
  verifier.
- Generic maps may omit or delete the standard `@payload` coordinate. UnixFS
  continues to require it as a layout invariant.
- `graph/writer` and the explicit resolver consume narrow ArcTable interfaces
  instead of importing concrete runtime packages. ArcTable remains untrusted
  execution state.
- `layout/unixfs` exposes separate Reader and Writer facades with explicit CAS
  capabilities. The compatibility `New` constructor remains available but is
  deprecated.
- `runtimegraph.NewGraph` no longer accepts an unused CAS argument.
- MIP-1011 and the reference specs define the experimental `v0alpha1` typed
  read/result and ProofList binding profile.
- Typed-root decoding and KZG proof verification fail closed on invalid
  commitment lengths, out-of-range proof indices, and non-canonical proof
  lengths.

## Compatibility And Limits

- All new facade names, query labels, ProofList fields, root codecs, and
  verifier interfaces remain experimental before `v1.0.0`.
- ProofList and resolve JSON do not have stable named JSON Schema files. The
  current envelope also has no embedded version discriminator.
- Removing the generic mandatory-`@payload` rule changes behavior for callers
  that previously treated `arcset.ErrMissingPayloadBinding` or writer payload
  errors as universal map validation.
- The reference daemon and CLI remain development/evaluation adapters. Managed
  tenancy, authorization, root publication, freshness, quotas, billing,
  pinning, garbage collection, and backend operations belong outside core.
- ProofList verification authenticates range metadata and segment CIDs. UnixFS
  callers accepting returned range bytes must also use
  `layout/unixfs.VerifyRangeBody` or an equivalent body-binding check.
- Current measured-list evidence uses fixed-width chunks. Variable-width
  measured lists remain future work.
- The KZG batch API currently serializes and verifies one primitive proof per
  index. A native multi-opening proof remains a post-release optimization.

## Release Validation

The release tree was validated from the repository root with:

```bash
git diff --check
go test ./...
go vet ./...
mkdir -p bin
go build -buildvcs=false -o bin/cas ./cmd/cas
go build -buildvcs=false -o bin/malt ./cmd/malt
go build -buildvcs=false -o bin/malt-eval ./cmd/eval/malt-eval
bin/malt-eval run --plan examples/eval-smoke-plan.json --run-id v0.0.3-smoke
```

Also verify:

- `auth/verifier` accepts runtime-generated KZG and IPA map/list/range proofs
  and rejects target, query, and range tampering.
- a relation-only generic map can be committed and verified without
  `@payload`.
- UnixFS read/write and range-body tests still enforce its payload bindings.
- production import guards keep operational layers outside `auth` and concrete
  runtime/storage packages outside `graph`.
- a temporary external Go module can import the root `malt` facade and
  `auth/verifier` without importing `server`.
- the CLI proof smoke below succeeds against isolated daemon state.
- malformed typed roots and proof-carried KZG indices are rejected without
  truncation or panic; the primitive proof parser is covered by negative and
  fuzz tests.

## CLI Proof Smoke

```bash
set -euo pipefail

SMOKE_DIR=$(mktemp -d)
MALT_CONFIG="$SMOKE_DIR/malt.json"
MALT_STATE="$SMOKE_DIR/malt-state"
MALT_LISTEN="${MALT_LISTEN:-127.0.0.1:14317}"
CAS_HOME="$SMOKE_DIR/cas-home"
CAS_CONFIG="$SMOKE_DIR/cas/settings.json"
CAS_DATA="$SMOKE_DIR/cas/data"
CAS_LISTEN="${CAS_LISTEN:-127.0.0.1:14318}"

cleanup() {
  bin/malt --config "$MALT_CONFIG" stop >/dev/null 2>&1 || true
  HOME="$CAS_HOME" bin/cas --config "$CAS_CONFIG" stop >/dev/null 2>&1 || true
  rm -rf "$SMOKE_DIR"
}
trap cleanup EXIT

mkdir -p "$CAS_HOME" "$SMOKE_DIR/cas"
HOME="$CAS_HOME" bin/cas --config "$CAS_CONFIG" init --non-interactive --force \
  --listen "$CAS_LISTEN" \
  --data-dir "$CAS_DATA"
HOME="$CAS_HOME" bin/cas --config "$CAS_CONFIG" start
bin/malt --config "$MALT_CONFIG" init --non-interactive --force \
  --state-root "$MALT_STATE" \
  --listen "$MALT_LISTEN" \
  --cas-base-url "http://$CAS_LISTEN"
bin/malt --config "$MALT_CONFIG" start
printf 'hello malt\n' >"$SMOKE_DIR/malt-v003-hello.txt"
ROOT=$(bin/malt --config "$MALT_CONFIG" add "$SMOKE_DIR/malt-v003-hello.txt" | awk '/Result root:/ {print $3}')
bin/malt --config "$MALT_CONFIG" resolve "$ROOT" malt-v003-hello.txt >"$SMOKE_DIR/malt-v003-resolve.json"
bin/malt --config "$MALT_CONFIG" verify --prooflist "$SMOKE_DIR/malt-v003-resolve.json"
```

The smoke owns the temporary state and ports it creates. If either port is in
use, override `MALT_LISTEN` or `CAS_LISTEN` and record the chosen values in the
release validation notes.
